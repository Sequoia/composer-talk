Package Manager vs. Dependency Manager

  • Package Manager: installs packages globally and/or to be invoked independently
  • Dependency Manager: installs packages locally, relative to a project, for the project
from composer: By default it will never install anything globally. Thus, it is a dependency manager.

Why Use a Dependency Manager?

  • Easier installation of application/library
  • Fewer version conflicts
  • Find problems faster (esp. for platform dependencies)
  • More consistency & transparency
  • $$$
  • clone project & run a single command
  • two projects that rely on same lib, diff. versions
  • If you're installing a project that relies on bz2 to backup & archive data and server doesn't have it, find out when you install instead of when you look for backups and realize you don't have any (yes, contrived example)
  • quickly get an overview of what a package depends on
  • Job prospects, working on others' projects; Soon this will be required knowledge: Laravel/ZF/Sym shop -> need this skill. In 12-24mo if you're not using this, you're behind

Installing Composer

  • Requires PHP 5.3.2 (Released Dec 20, 2012)
  • Install via curl (*nix):
    $ curl -sS https://getcomposer.org/installer | php
  • Move into your $PATH for easier access 
    $ mv composer.phar /usr/local/bin/composer
# now reachable via `composer [command]
# rather than `php composer.phar [command]
  • Need late-version PHP
  • putting in path is optional, allows you to call composer rather than php composer.phar

composer.json

  • List package information
  • Define dependencies
  • Declare where the files for your package are
  • Constrain what types of dependencies are installed
  • ...and more
  • info: name, authors, license, repo location, where to get support
  • using psr-0 or other methods
  • minimum-stability & prefer-stable
{
    "name": "monolog/monolog",
    "description": "Sends your logs to files, sockets, inboxes, databases and various...",
    "keywords": ["log", "logging", "psr-3"],
    "homepage": "http://github.com/Seldaek/monolog",
    "type": "library",
    "license": "MIT",
    "authors": [
        {
            "name": "Jordi Boggiano",
            "email": "j.boggiano@seld.be",
            "homepage": "http://seld.be"
        }
    ],
    "require": {
        "php": ">=5.3.0",
        "psr/log": "~1.0"
    },
    "require-dev": {
        "mlehner/gelf-php": "1.0.*",
        "raven/raven": "0.5.*",
        "doctrine/couchdb": "dev-master"
    },
    "suggest": {
        "mlehner/gelf-php": "Allow sending log messages to a GrayLog2 server",
        "raven/raven": "Allow sending log messages to a Sentry server",
        "doctrine/couchdb": "Allow sending log messages to a CouchDB server",
        "ext-amqp": "Allow sending log messages to an AMQP server (1.0+ required)",
        "ext-mongo": "Allow sending log messages to a MongoDB server"
    },
    "autoload": {
        "psr-0": {"Monolog": "src/"}
    },
    "extra": {
        "branch-alias": {
            "dev-master": "1.6.x-dev"
        }
    }
}
  • name, description, keywords. used by packagist / composer search
  • re type: this can be library, project, metapackage, or composer-installer: custom installer can e.g. put files outside of the /vendor directory. Some frameworks use these for custom install logic
  • support is another great key with metadata: tell your uses where to file bugs, where the forum is, where the IRC channel is KNIT COMMUNITY TOGETHER
  • require: platform deps: php version & ext- lib- bcrypt or curl
  • require: other packages: version: *, ~, >= etc.
  • require-dev: tools used for dev
  • suggest: other packages that are useful with this one. also conflict & replace
  • autoload defines where sources for your package are, and how they're structured. psr-0 here: /vendor/namespace/library, defines how _ are used in path names etc.. Can also load non-psr-0 libs with classmap or include files on each request with `files`
  • extras here has branch-alias
  • NOT "version": git tracks this

Root-only properties

  • require-dev
  • minimum-stability
  • repositories
  • config
  • scripts
  • only work if what you're installing is the root project
  • require-dev: tools for testing, compiling css, etc.
  • minimum-stability: prod package you want all stable stuff, disallow deps from loading >stable libs
  • repositories: use this when you want to point to a repo other than packagist
  • config: where deps are installed, cache size, process-timeout, etc.
  • scripts: pre-install and post-install most notable. Can call static class methods directly so you can put install logic in a class in your app
  • SO THIS SEEMS TO WORK, allows easy install, but how do you ensure that your application will work after install? what if a minor lib change breaks something subtly? enter: composer.lock

composer.lock

  • Record which dependency versions are currently installed
  • Check this into your VCS (unless it's a library)
  • Exact same libs (from same location) across all servers
  • Clone & install 12 months later: all same lib versions
  • Update lockfile with composer update [packagename]
  • known working set of deps
  • update & run tests, if fail, revert to last committed lockfile
  • update deps together or one X one
  • only works on root project

Publishing to Packagist.org

  • You've got your great new library, ready to share with the world
  • How do you get your package on Packagist?
  • Just paste in your github repo location!

How Packagist Works

  • Gets package metadata from composer.json
  • Lists release versions based on tag names (or comparable branch name)
  • Users may also require branches directly using dev-branchname
  • Scans repo once per day for new code
  • Service hook for github can tell packagist to rescan whenver you push
  • comparable means countable, +1 able (basically, a number)
  • Use semantic versioning!!!

CI with Travis

Travis CI is a hosted continuous integration service for the open source community.
  • Runs when you push to github
  • Can test against multiple PHP versions
  • Notifications on build failure
  • Has common tools (xdebug, phpunit, composer)
  • .travis.yml tells travis how to run your build
  • Best part:
  • Continuous integration: building & testing at regular intervals rather than once after a long dev period
  • PHP isn't compiled but a package depending on other packages does require a build
  • Multiple php versions, great!
  • travis-ci.org

What about PEAR?

  • Only one channel for a long time
  • Language features (namespaces)
  • Frameworks
  • Many packages not updated to PHP5
  • Github filled many of the same needs, better
  • vs. Composer: PEAR requires server software, Composer can install from git repo
  • Successes: PEAR installer itself, DB, MDB2; many projects have channels still
  • Still used for: PHP extensions
  • PEAR the installer vs. PEAR the repo
  • Only one channel: quality over quantity, vetted packages, hard to grow
  • ZEND framework, "official php libarary" 1 big chunk, didn't use PEAR. others followed
  • By the time they added channels, hard to set up, people already on frameworks; package spec undocumented
  • no namespaces -< hard to share code -< large frameworks instead
  • PHP5: devs didn't see need to update, many maintainers left project & didn't get replaced
  • github: getting/hosting code, issue tracking,
  • Pear installer still widely used; ESP. for PHP Extensions
